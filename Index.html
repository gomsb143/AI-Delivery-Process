01
Program & Project Initiation
Owner: Program Manager · Director
▼

Align AI projects with business priorities. Establish the charter, secure sponsorship, assign cross-functional teams, and build a high-level roadmap before any technical work begins.

⬇ Inputs

  • Business goals & strategic priorities
  • Portfolio requests / backlog
  • Available budget & headcount

⬆ Outputs

  • Project charter document
  • High-level roadmap & milestones
  • Resource & team allocation plan

Key Activities

  • Portfolio review with executive stakeholders
  • Define project objectives, scope, and success criteria
  • Assign cross-functional team leads: Data Science, DevOps, Compliance
  • Weekly executive checkpoints scheduled

⚠ Risk

Misaligned priorities between business units. AI project scoped incorrectly vs. strategic goals.

✓ Mitigation

Executive alignment workshops. Prioritize high-value projects using a weighted scoring matrix.

⚡ Workaround

If alignment stalls, set flexible milestone buffers and re-run priority scoring with updated business input.

▸ Real-World Scenarios

Budget Conflict
Late Sponsorship
Scope Creep Risk

Scenario: Multiple teams competing for the same AI budget

Two business units submit AI project requests exceeding the total portfolio budget by 40%. The PM must facilitate prioritization.

  1. Run a weighted scoring workshop: ROI potential (40%), strategic fit (30%), feasibility (30%)
  2. Present ranked projects to the Program Director with clear tradeoff analysis
  3. Phase lower-ranked projects into a future sprint or propose a shared resource model
  4. Document decisions in the charter and circulate to all stakeholders

Scenario: Executive sponsor unavailable at kickoff

The designated executive sponsor is on leave. The project charter cannot be signed off, blocking team allocation.

  1. Identify a temporary delegated authority within 48 hours
  2. Issue a provisional charter with a 2-week review window
  3. Begin low-risk activities: team onboarding, requirements prep
  4. Formally ratify charter upon sponsor's return

Scenario: Stakeholders adding features mid-charter

After charter sign-off, three new AI use-cases are submitted by business teams requesting inclusion.

  1. Log incoming requests in a formal change request register
  2. Assess impact on timeline, cost, and resource allocation
  3. Present to Program Director: accept into current scope or backlog
  4. Update charter only if approved; communicate changes to all teams
02
Business Requirements & Functional Analysis
Owner: Functional Consultants · PM
▼

Define precisely what the AI solution must achieve. Capture KPIs, edge cases, integration requirements, and data availability constraints before handing to architects.

⬇ Inputs

  • Business goals from charter
  • Stakeholder interviews & surveys
  • Current process flow documentation
  • Data availability assessment

⬆ Outputs

  • Functional requirements document
  • KPIs and measurable success metrics
  • MVP feature set definition
  • Prototype or mock dataset

Key Activities

  • Stakeholder workshops: capture pain points and desired outcomes
  • Map current-state vs. future-state process flows
  • Define edge cases and exception handling criteria
  • Iterative review cycles with prototype datasets

⚠ Risk

Requirements incomplete or change frequently due to evolving business understanding of AI capabilities.

✓ Mitigation

Iterative requirement reviews with business sign-off. Include prototype datasets to validate feasibility early.

⚡ Workaround

Tight deadline: Deliver MVP requirements first, capture remaining requirements as backlog items for Sprint 2.

▸ Real-World Scenarios

Changing Requirements
Data Unavailable
KPI Disagreement

Scenario: Business changes requirements after sign-off

After functional spec is approved, a new regulatory requirement shifts the target outcome by 30%.

  1. Freeze the current spec and open a formal change request
  2. Impact-assess: which model features, data pipelines, and KPIs are affected?
  3. Negotiate: defer non-critical changes to Phase 2 release
  4. Update requirements doc with version history and stakeholder sign-off

Scenario: Required training data doesn't exist yet

The AI model needs 18 months of historical transaction data, but the system only has 6 months available.

  1. Flag data gap to Data Science team immediately — do not proceed silently
  2. Evaluate synthetic data generation as a temporary training supplement
  3. Adjust success metrics to reflect model limitations during initial deployment
  4. Plan a model refresh once sufficient real data accumulates (6–12 months)

Scenario: Two business units define success differently

Marketing wants precision, Operations wants recall. A single model cannot perfectly optimize both simultaneously.

  1. Facilitate a joint workshop to explain the precision/recall tradeoff visually
  2. Define a composite F1-score target as a shared KPI
  3. Document each unit's secondary metric for monitoring purposes
  4. Escalate to Program Director if no consensus is reached within 1 sprint
03
Solution Architecture & Design
Owner: Solution Architects · PM
▼

Translate functional requirements into a technically feasible AI architecture. Design data pipelines, integration points, and a modular system that can scale with model iterations.

⬇ Inputs

  • Functional requirements document
  • Available tech stack & cloud platform
  • Data source map & API inventory

⬆ Outputs

  • High-level architecture diagram
  • Integration & API plan
  • Data pipeline design
  • PoC prototype decision

Key Activities

  • Design modular, decoupled architecture to allow model swaps
  • Run early proof-of-concept (PoC) to validate feasibility
  • Define data pipeline stages: ingestion → processing → serving
  • Document integration points with existing enterprise systems

⚠ Risk

Architecture not feasible for AI scale. On-prem infrastructure unable to handle model training loads.

✓ Mitigation

Early PoC; modular architecture so components can be replaced. Cloud-burst capability for compute spikes.

⚡ Workaround

Use cloud-based AI services (Azure ML, AWS SageMaker, GCP Vertex) for short-term deployment while on-prem is upgraded.

▸ Real-World Scenarios

Infra Limitation
Legacy Integration
PoC Failure

Scenario: On-prem servers can't handle model training load

During PoC, GPU utilization hits 100% for 14-hour training runs, blocking other workloads.

  1. Immediately flag infrastructure bottleneck to Program Director
  2. Evaluate cloud burst strategy: spot instances on AWS or Azure for training only
  3. Serve inference on-prem using optimized/quantized model to reduce compute
  4. Long-term: include GPU upgrade in next budget cycle

Scenario: Critical data locked in a legacy system with no API

The most valuable training data is in a 15-year-old ERP system with no REST API and unreliable ODBC drivers.

  1. Assess: scheduled batch export vs. real-time stream vs. full migration
  2. Implement a CDC (Change Data Capture) layer using Debezium or similar
  3. Build a data staging layer (data lake) as the integration intermediary
  4. Flag ongoing maintenance cost of the integration to the PM and architect

Scenario: PoC results are inconclusive — should we proceed?

The PoC model achieves 61% accuracy against a target of 80%. The team is split on whether to proceed.

  1. Diagnose root cause: data quality, feature engineering, or model choice?
  2. Run a second PoC iteration with improved features (2-week timebox)
  3. If still under threshold: propose a simpler rule-based fallback system
  4. Present go/no-go recommendation to Program Director with data to support decision
04
Risk & Governance Assessment
Owner: Risk / Compliance · PM
▼

Ensure legal compliance, reduce model risks, and create audit trails before production. Address bias, privacy, and explainability requirements upfront — not after deployment.

⬇ Inputs

  • Solution architecture design
  • Regulatory requirements (GDPR, etc.)
  • Business requirements & KPIs
  • Data classification report

⬆ Outputs

  • Risk register (scored & prioritized)
  • AI governance plan
  • Data privacy impact assessment
  • Audit trail specification

Key Activities

  • Run bias detection analysis on training datasets
  • Engage legal/compliance early — do not wait until deployment
  • Define model explainability requirements (LIME, SHAP)
  • Create audit trail and logging specification

⚠ Risk

Regulatory non-compliance (GDPR, CCPA). Biased outcomes in protected groups. Unexplainable model decisions.

✓ Mitigation

Bias detection tools (Fairlearn, IBM AIF360). Synthetic or anonymized training data. Early legal engagement.

⚡ Workaround

Document known compliance gaps. Deploy with human-in-the-loop oversight as temporary measure until gaps resolved.

▸ Real-World Scenarios

GDPR Conflict
Model Bias Found
No Explainability

Scenario: Model uses PII data that violates GDPR Article 22

The model was designed to use customer demographic features for scoring, which triggers automated decision-making restrictions under GDPR.

  1. Immediately halt model training on the affected features
  2. Engage legal counsel to clarify permissible feature scope
  3. Re-engineer features using anonymized proxies or aggregated signals
  4. Add human review checkpoints to automated decisions as interim safeguard

Scenario: Bias detected against a protected demographic

Fairlearn analysis shows the model has a 23% lower approval rate for a protected group compared to a reference group with similar credit profiles.

  1. Halt production readiness until root cause is found
  2. Audit training data for historical bias and class imbalance
  3. Apply re-weighting or adversarial debiasing techniques
  4. Re-validate with fairness metrics before re-submitting for governance approval

Scenario: Business demands explainability but model is a black box

A neural network achieves target accuracy, but regulators require per-decision explanations. The model provides none.

  1. Implement SHAP values for post-hoc explanations on each prediction
  2. Evaluate simpler surrogate model (decision tree) as a parallel explainability layer
  3. Present top-3 feature contributions per decision in the UI
  4. Document explainability approach in the governance plan for regulator review
05
Data Science & Model Development
Owner: Data Scientists · PM
▼

Build, test, and validate AI models iteratively. This phase loops — experiments run until KPIs are met. Maintain parallel experiments and document all assumptions.

🔁 ITERATION NOTE  This phase is non-linear. Multiple experiments run simultaneously. PM tracks each experiment branch and gates production based on KPI thresholds, not time alone.

⬇ Inputs

  • Cleaned & labeled datasets
  • Feature engineering specifications
  • Business KPIs & success thresholds
  • Governance constraints (from Phase 4)

⬆ Outputs

  • Trained model artifact(s)
  • Evaluation metrics & test reports
  • Experiment logs (MLflow, W&B)
  • Champion model selection rationale

Key Activities

  • Maintain multiple model experiments in parallel (champion-challenger)
  • Feature engineering iterations based on performance analysis
  • Cross-validation and holdout set testing
  • Document all assumptions, hyperparameters, and experiment decisions

⚠ Risk

Poor model performance. Insufficient training data. High variance / overfitting. Experiment results not reproducible.

✓ Mitigation

Champion-challenger experiments. Simpler fallback models. Feature engineering iterations. Reproducible experiment tracking with MLflow.

⚡ Workaround

Deploy pre-trained foundation model or rules-based interim system to meet deadline. Schedule model improvement for next sprint.

▸ Real-World Scenarios

Model Fails KPI
Data Quality
Deadline Pressure

Scenario: After 6 iterations, model still misses KPI by 12%

The target F1 score is 0.85. After 6 experiment cycles, the best model achieves 0.75. Timeline is at risk.

  1. Conduct a root cause analysis: data quality, feature set, or model architecture?
  2. Bring in an external ML consultant for a fresh perspective (2-day timebox)
  3. Negotiate with business: can the KPI threshold be adjusted for Phase 1?
  4. If KPI cannot be met: deploy a rules-based fallback and schedule model V2

Scenario: Training data has significant label noise

Data audit reveals ~18% of labeled samples are incorrectly classified, causing the model to learn incorrect patterns.

  1. Halt model training — garbage in, garbage out. Alert PM immediately.
  2. Run label quality tools (Cleanlab, confidence learning) to identify noisy samples
  3. Engage subject matter experts for re-labeling the suspect subset
  4. Retrain from clean dataset and update timeline estimates accordingly

Scenario: Business demands deployment in 3 weeks, model needs 3 months

An executive deadline cannot be moved. The current model development timeline exceeds it by 10 weeks.

  1. Deploy a pre-trained model (GPT, BERT, XGBoost base) as a fast MVP
  2. Fine-tune the pre-trained model on available data in a 2-week sprint
  3. Set clear expectations: MVP performance vs. target KPI — communicate gap in writing
  4. Full custom model development continues in parallel for Q2 release
06
Deployment / MLOps
Owner: Solution Engineering · DevOps
▼

Integrate validated models into production with automated pipelines, rollback plans, and canary releases. Zero-downtime deployment is the goal — not just getting the model live.

⬇ Inputs

  • Validated champion model artifact
  • Integration plan from Phase 3
  • Governance approval (Phase 4)
  • Performance benchmarks

⬆ Outputs

  • Production-ready AI system
  • CI/CD pipelines for model updates
  • Automated testing suite
  • Rollback & incident playbooks

Key Activities

  • Canary release: route 5% of traffic to new model before full rollout
  • Shadow mode: run new model alongside existing system without acting on predictions
  • Automated regression testing on model outputs
  • Document rollback trigger conditions and procedures

⚠ Risk

Deployment failures, integration breaks, scaling issues under production load, model serving latency spikes.

✓ Mitigation

Canary releases. Monitoring dashboards from Day 1. Automated rollback triggered by latency or error thresholds.

⚡ Workaround

Staged deployment or shadow mode operation until production issues are resolved. Previous version remains active.

▸ Real-World Scenarios

Deployment Failure
Latency Spike
Rollback Trigger

Scenario: Container image fails to start in production

During canary deployment, the model container crashes due to a missing environment variable in production config.

  1. Auto-rollback fires immediately — canary traffic reverts to stable model
  2. DevOps investigates config diff between staging and production environments
  3. Add environment variable validation to CI/CD pre-deployment checklist
  4. Re-deploy with fixed config after passing staging smoke tests

Scenario: Model inference latency triples after load increase

At 10x expected traffic, model inference p99 latency hits 4.2 seconds vs. 400ms SLA target.

  1. Immediate: enable model response caching for high-frequency identical inputs
  2. Quantize model (INT8) to reduce compute requirement — often 2-4x speed improvement
  3. Scale horizontal replicas via Kubernetes HPA based on request queue depth
  4. Long-term: profile bottleneck (GPU, network, serialization) and optimize

Scenario: When should PM trigger a rollback?

PM receives alerts but is unsure whether to rollback or wait. Clear criteria prevent panic decisions.

  1. Pre-define rollback triggers: error rate >2%, latency p99 >1s, accuracy drop >5%
  2. Rollback decision is automatic if alerts breach thresholds for more than 5 minutes
  3. PM notifies business stakeholders within 15 minutes of rollback event
  4. Post-incident review within 48 hours — document root cause and prevention steps
07
Monitoring & Continuous Improvement
Owner: MLOps · Monitoring Team
▼

Ensure the AI model stays relevant and effective as real-world data evolves. Monitor for concept drift, automate retraining, and maintain human override capabilities.

⬇ Inputs

  • Production inference logs
  • User feedback & outcome signals
  • KPI dashboards & alerts
  • New incoming training data

⬆ Outputs

  • Monitoring dashboards & alerts
  • Scheduled retraining pipeline
  • Model performance reports
  • Improvement backlog

Key Activities

  • Continuous drift detection on input data distribution vs. training distribution
  • Automated retraining pipeline triggered by performance thresholds
  • Monthly performance review with business stakeholders
  • Maintain manual override capability for high-risk decisions

⚠ Risk

Concept drift — real-world data shifts away from training distribution. Model degradation undetected until business impact is felt.

✓ Mitigation

Scheduled retraining (weekly/monthly). PSI/KS-test drift alerts. Performance threshold alerts with escalation workflow.

⚡ Workaround

Temporary fallback to rules-based system while retraining completes. Human-in-the-loop review for low-confidence predictions.

▸ Real-World Scenarios

Concept Drift
Silent Failure
Retraining Cost

Scenario: Model degrades after a market shift

6 months post-deployment, a regulatory change alters customer behavior patterns. Model accuracy drops from 83% to 67% over 3 weeks.

  1. PSI drift alert fires at week 2 — PM and Data Science notified automatically
  2. Activate human-in-the-loop review for predictions with confidence < 70%
  3. Trigger emergency retraining pipeline with last 3 months of post-change data
  4. Validate retrained model in staging before promoting; communicate timeline to business

Scenario: Model appears to be working but outcomes are wrong

Accuracy metrics show 81% but business reports a 35% increase in customer complaints linked to AI decisions.

  1. Recognize that accuracy alone is insufficient — add business outcome metrics to dashboard
  2. Investigate: are low-confidence correct predictions causing downstream issues?
  3. Add calibration metrics and business outcome KPIs to the monitoring suite
  4. Establish feedback loop: capture complaint outcomes as ground truth labels for retraining

Scenario: Retraining pipeline costs escalate unexpectedly

Monthly retraining costs reach $40K due to full dataset retraining each cycle. Budget only supports $8K/month.

  1. Switch to incremental/online learning — train only on new data deltas
  2. Implement warm-start retraining from the previous model checkpoint
  3. Adjust retraining frequency based on drift severity: monthly by default, weekly only when drift detected
  4. Evaluate lighter model architecture that achieves acceptable accuracy at lower cost

RACI Governance Matrix

Role Definitions

Program Manager

Accountable across all phases. Owns timeline, cross-team coordination, escalation, and stakeholder communication. Single thread of program ownership.

Data Science

Owns model experiments, feature engineering, and evaluation. Consulted on requirements and architecture. Reports to PM on experiment status weekly.

Solution Architects

Responsible for Phase 3. Consulted on Phases 4 and 6. Ensures the architecture can support model updates without re-engineering the entire system.

Risk / Compliance

Responsible for Phase 4. Must be engaged before Phase 3 completes to prevent governance blockers at deployment. Signs off on production readiness.

Risk Register

Consolidated risk inventory across all 7 phases. Review at every weekly PM checkpoint. Escalate HIGH risks to Program Director within 24 hours of identification.

⚠ ESCALATION RULE  Any HIGH risk that has no mitigation owner assigned within 48 hours auto-escalates to Program Director. Document all risks in your PM tool (Jira Epic / Asana portfolio risk field).
